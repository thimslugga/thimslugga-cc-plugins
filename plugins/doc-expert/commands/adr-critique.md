# /adr-critique

Use this command when you want to audit one ADR (or a corpus of them) that wasn't drafted via this plugin's push-back flow — legacy ADRs, ADRs imported from other teams, or ADRs generated by generic LLM tooling.

## What this command does

Hands the audit to the `doc-expert` agent, which activates the `adr-critique` skill. The skill:

1. **Reads the target ADR and its neighbors** — follows `supersedes`, `amends`, `relates-to` links to gather graph context.
2. **Flags line by line** — quotes the offending line verbatim, names the rule broken (from the shared "ADR is NOT" checklist), proposes a shorter rewrite, asks for per-line approval. One flag per message.
3. **Runs the missing-why check** — every line of the Decision section must tie a business concern to an architectural characteristic. "Industry standard" / "best practice" / "most teams" trigger flags.
4. **Runs the consistency check** — bidirectional supersession, tensions, duplicate authority, ghost references.
5. **Runs the LikeC4 drift check** — name and relationship mismatches between the ADR and any `.c4` files.
6. **Applies approved changes in one pass** — touches only architect-approved lines, no audit markers.

## Your input

- The path to the ADR (or directory) to audit
- Optionally: whether you want the audit to include the LikeC4 drift check (default yes if any `.c4` files exist)

## What this command will NOT do

- Bulk-edit. Every change requires per-line approval.
- Touch an `accepted` ADR's body. The audit can recommend supersession but cannot rewrite an accepted ADR in place — that has to go through `adr-drafting` as a new ADR.
- Add `[reviewed]` stamps, dates, or signatures into the file.
- Decide which side wins when LikeC4 and the ADR drift — surfaces the mismatch, you choose.

Use `/doc-audit` instead when the goal is folder-level classification (KEEP / MERGE / REWRITE / DELETE / MOVE). Use `/adr-critique` for deep per-line audit of an ADR you want to keep but tighten.
